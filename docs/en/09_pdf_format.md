# Expected PDF Format

## Description

The `vigilant/parsers/pdf_parser.py` module is designed to extract metadata from PDF reports automatically generated by proprietary CCTV systems. This document describes the expected format.

## Expected Structure

### Main Fields (Header)

The PDF must contain the following fields anywhere in the document:

```
Usuario: <username>
Fecha de inicio: <process_start_date>
Fecha de finalización: <process_end_date>
Tipo: <process_type>
Dividir archivo: <Yes/No>
Firmar: <Yes/No/Sí>
Marca de agua: <Yes/No/Sí>
```

### Recording Sections

Each recording is identified by the marker `Nombre del canal:` followed by its metadata:

```
Nombre del canal: <name>
Fecha de inicio: <recording_start_date>
Fecha de finalización: <recording_end_date>
Estado: <status>
- <generated_file_path>
```

## Error Handling

### Robust Behavior

As of version 0.2.0, the parser implements robust error handling:

- **Missing fields**: If a field is not found, a default value is used.
- **Unexpected format**: If the PDF opens but does not contain the expected format, it returns a structure with defaults.
- **Unreadable/Cannot open PDF**: The CLI logs the error and continues with the next file (it does not abort the batch).
- **Empty blocks**: These are automatically ignored.

### Default Values

| Field | Default |
|-------|---------|
| `username` | `"Unknown"` |
| `process_start_date` | `""` (empty string) |
| `process_end_date` | `""` |
| `tipo` | `""` |
| `dividir_archivo` | `""` |
| `firmar` | `""` |
| `marca_agua` | `""` |
| `nombre_canal` | `""` |
| `fecha_inicio` (recording) | `""` |
| `fecha_fin` (recording) | `""` |
| `status` | `""` |

### CLI Usage

```bash
# Parse PDFs to JSON
vigilant parse --input-dir data/pdf --output-dir data/json
```

## Path Normalization

Paths in `generated_file` are automatically normalized:
- Backslashes (`\`) are converted to forward slashes (`/`)
- Example: `C:\Videos\file.mp4` → `C:/Videos/file.mp4`

## Technical Notes

### Text Extraction (Internal Implementation)

The parser uses PyMuPDF (fitz) to extract all text from the PDF:

```python
doc = fitz.open(str(pdf_path))
text = "\n".join(page.get_text() for page in doc)
```

### Regex Patterns

Patterns are not case-sensitive and search for the field followed by any content until the end of the line:

- User: `r"Usuario: (.+)"`
- Dates: `r"Fecha de inicio: (.+)"` and `r"Fecha de finalización: (.+)"`
- Type: `r"Tipo: (.+)"`
- Status: `r"Estado: (.+)"`

### Security

The parser **DOES NOT validate** or **sanitize** the extracted content. It is assumed that the PDFs come from trusted CCTV system sources.

> [!WARNING]
> Do not use this parser with PDFs from untrusted sources without additional validation.

## Test Cases

To validate that a PDF is compatible, check that:

1. It contains at least the field "Usuario:".
2. Dates are in a readable format (they are not parsed, only extracted as strings).
3. Each recording has the line "Nombre del canal:".
4. Generated file paths start with "- ".

## Troubleshooting

### Problem: User shows as "Unknown"

**Possible causes:**
- The PDF does not contain the line `Usuario: <name>`.
- There are special characters or incorrect encoding.
- The pattern does not match exactly.

**Solution:**
Verify that the PDF contains exactly `Usuario:` followed by a space and the name.

### Problem: Empty recordings

**Possible causes:**
- The PDF does not contain "Nombre del canal:".
- The blocks are empty or malformed.

**Solution:**
Each recording section must start explicitly with `Nombre del canal:`.
